**Запустите платформу Citeck — на своём компьютере или на сервере — из одного бинарника.**

[Citeck](https://github.com/Citeck) — self-hosted open-source альтернатива проприетарным ECM/BPM-системам: управление документами и записями, автоматизация маршрутов согласования во встроенном дизайнере BPMN, создание внутренних приложений — порталов, CRM, управления обращениями — почти без кода или вообще без него. Пользователи, роли и права доступа уже встроены.

Запускать это вручную — значит оркестрировать пару десятков Docker-сервисов. Citeck Launcher делает это за вас: единый бинарник ~24 МБ устанавливает платформу, запускает каждый сервис (Keycloak, PostgreSQL, RabbitMQ и веб-приложения Citeck) как Docker-контейнер, следит за их работоспособностью и обновляет их. Это штатный способ запуска Citeck — как десктопное приложение или из командной строки на сервере.

**Что понадобится:** Docker · **16 ГБ** RAM для редакции Community, **24–32 ГБ** для Enterprise (~24 сервиса) · **50+ ГБ** свободного места на диске под образы и данные. На Windows и macOS сначала установите [Docker Desktop](https://www.docker.com/products/docker-desktop/).

## Десктоп или сервер?

Есть два способа запуска — выберите тот, что соответствует тому, **где** вы хотите запустить Citeck:

| | 🖥 **Десктопное приложение** | 🖧 **Сервер (CLI)** |
|---|---|---|
| Для чего | Ваш собственный компьютер | Linux-сервер / ВМ (обычно по SSH) |
| Установка | Скачать установщик, пройти мастер | Одна команда `curl … \| bash` |
| Интерфейс | Нативное окно приложения (GUI) | Терминал — CLI `citeck` + мастер настройки |
| Начать здесь | [Десктопное приложение](#десктопное-приложение) | [Установка на сервер](#установка-на-сервер) |

> **Обратите внимание:** быстрый старт `curl … | bash` и команды `citeck` в этом README предназначены для **установок на сервер**. На своём компьютере запускайте Citeck через **десктопное приложение** — там всё делается из интерфейса.

## Десктопное приложение

Десктопное приложение запускает Citeck на вашем компьютере с Windows, macOS или Linux — обычное окно приложения, без командной строки. Citeck продолжает работать в фоне даже после закрытия окна.

Сначала установите Docker Desktop, затем скачайте установщик для своей платформы из [последнего релиза](https://github.com/Citeck/citeck-launcher/releases/latest):

| ОС | Файл | Архитектуры |
|----|------|-------------|
| Windows | `citeck-desktop_<version>_windows_<arch>.msi` | amd64, arm64 |
| macOS | `citeck-desktop_<version>_darwin_<arch>.dmg` | amd64 (Intel), arm64 (Apple Silicon) |
| Linux | `citeck-desktop_<version>_linux_<arch>.deb` / `.rpm` | amd64, arm64 |

К каждому установщику прилагается файл `.sha256` для проверки. Ваши данные сохраняются при обновлениях.

## Установка на сервер

> **Для Linux-сервера или ВМ** (amd64 или arm64) — выполняйте эти шаги на сервере, по SSH. Требование: Docker установлен и запущен.

```bash
curl -fsSL https://github.com/Citeck/citeck-launcher/releases/latest/download/install.sh | bash
```

Скрипт скачивает последний релиз для вашей платформы, устанавливает его в `/usr/local/bin/citeck` и запускает мастер настройки (`citeck install`). Мастер **интерактивный и требует настоящего терминала**. Он спросит:

- **доменное имя или IP**, по которому вы будете открывать платформу в браузере;
- как **защитить соединение** — автоматически, Let's Encrypt, самоподписанный сертификат, свой собственный сертификат или обычный HTTP. (Для Let's Encrypt нужны публичное DNS-имя, указывающее на этот хост, и открытый входящий порт 80; если хост недоступен, мастер откатится к самоподписанному сертификату.)
- разворачивать ли **демо-данные** и устанавливать ли **systemd-сервис**.

### Первый запуск: чего ожидать

**Это займёт время — так и должно быть.** Лаунчер скачивает несколько ГБ Docker-образов, а затем самой платформе нужно примерно **10–15 минут**, чтобы подняться: сервисы стартуют в порядке зависимостей, а Keycloak при первом запуске импортирует свой realm. Наблюдайте, как приложения одно за другим переходят в `RUNNING`:

```bash
citeck status -w
```

Когда всё поднимется, мастер выведет данные для доступа:

```
Citeck is ready!

Open in browser:  https://<the domain you entered>/
Login:            admin / <generated password>
```

Про этот экран важно знать две вещи:

- **Пароль администратора показывается один раз.** Скопируйте его — восстановить его потом нельзя. Если потеряли, сбросьте через `citeck setup admin-password`.
- **С самоподписанным сертификатом браузер выдаст предупреждение.** Это ожидаемо — нажмите *Дополнительно* → *Перейти*.

Если через ~20 минут что-то по-прежнему выглядит зависшим, начните с `citeck diagnose` (добавьте `--fix`, чтобы лаунчер починил, что сможет) и `citeck logs <app>`.

### Обновление лаунчера

Выполните тот же one-liner ещё раз — скрипт определит установленную версию, спросит подтверждение на обновление, остановит демон и заменит бинарник. Предыдущий бинарник сохраняется в `/usr/local/bin/citeck.bak` и восстанавливается через `citeck install --rollback`. Ваши данные сохраняются.

## Понятия

Три слова, которые встречаются по всему CLI и документации:

- **Namespace** (неймспейс) — один изолированный экземпляр платформы (свои контейнеры, тома и данные). К неймспейсам Linux или Kubernetes отношения не имеет; это понятие лаунчера. На типичном сервере он ровно один.
- **Bundle** (бандл) — набор приложений и их версий, составляющих релиз платформы, например релиз Community или Enterprise. Переключение между ними — `citeck upgrade <bundle:version>`.
- **Workspace** (воркспейс) — откуда берутся эти определения (обычно Git-репозиторий или офлайн-архив `.zip` для установок без доступа в интернет).

## Повседневные команды (серверный режим)

В десктопном режиме те же операции доступны из интерфейса приложения.

```bash
citeck status -w                 # следить за неймспейсом и всеми приложениями
citeck logs <app> -f             # стримить логи (без приложения — лог самого демона)
citeck stop <app>                # остановить приложение — и не поднимать его при перезапусках
citeck start <app>               # снова запустить его (вернуть под управление)
citeck reload                    # применить изменения конфигурации, пересоздать только изменившееся
citeck snapshot export <name>    # бэкап всех томов (останавливает платформу, затем запускает её снова)
citeck upgrade <bundle:version>  # переключиться на другую версию платформы
citeck diagnose --fix            # проверки состояния с опциональным авто-восстановлением
citeck setup                     # изменить настройки (пароль администратора, TLS, почта, ресурсы…)
citeck edit <app>                # править определение приложения — в стиле kubectl edit
```

Учтите, что `citeck stop <app>` **отцепляет** (detach) приложение: оно остаётся остановленным при перезапусках и перезагрузках конфигурации, пока вы не выполните `citeck start <app>`. Это же способ освободить память на небольшом хосте — отцепив несколько необязательных приложений, вы сэкономите несколько ГБ.

Глобальные флаги: `--format (text|json)` для скриптов, `--yes/-y` чтобы пропустить подтверждения, `-d/--detach` чтобы вернуть управление сразу, не дожидаясь завершения. Полный справочник: `citeck --help` или [справочник команд](https://citeck-ecos.readthedocs.io/ru/latest/admin/launch_setup/launcher_server/commands.html).

## Что вы получаете

- **Самовосстанавливающийся рантайм** — liveness-пробы перезапускают упавшие сервисы, а лаунчер фиксирует причину падения
- **Резервное копирование и восстановление** — экспорт всех томов в один архив и импорт обратно на этот же или другой хост
- **HTTPS из коробки** — Let's Encrypt с авто-обновлением (домены *и* IP-адреса) или ваш собственный сертификат
- **Состояние и логи в реальном времени** — потребление ресурсов и потоковые логи каждого сервиса, в десктопном приложении или в CLI
- Локализация на 8 языков и автодополнение для bash, zsh, fish и PowerShell

## Редакции

Редакция **Community** полностью открытая и бесплатная, она покрывает основной функционал платформы. Коммерческая редакция **Enterprise** добавляет профессиональную поддержку и дополнительный функционал; для её установки нужен лицензионный ключ, выданный Citeck. Лаунчер устанавливает любую из них.

## Модель безопасности

Скажем сразу: **демон в серверном режиме управляет Docker, поэтому его API стоит считать эквивалентом root-доступа на хосте** (например, `citeck exec` выполняет команды внутри контейнеров). Именно поэтому по умолчанию выбран безопасный вариант.

- **CLI** общается с демоном через Unix-сокет, доступный только пользователю демона (права 0600).
- **Собственный Web UI лаунчера в серверном режиме по умолчанию выключен** — поддерживаемый интерфейс сервера это CLI/TUI. Когда вы его включаете (`server.webui.enabled: true` в `daemon.yml`), демон дополнительно слушает TCP-порт. При привязке к localhost полный API защищён только от браузерного CSRF — а это **не** аутентификация, поэтому любой локальный пользователь или процесс с доступом к порту получает полный контроль. Включайте осознанно и только на **выделенном (single-tenant) хосте**, все локальные пользователи которого и так имеют доверенный доступ уровня Docker/root. Привязка к внешним адресам требует клиентских сертификатов mTLS.
- **Чтобы закрыть этот пробел на localhost**, включите токен-аутентификацию API: `api_auth.enabled: true` в `daemon.yml`. После этого каждый запрос к `/api` по TCP требует `Authorization: Bearer <token>` (или сессионную cookie браузера, выдаваемую через `GET /auth/session?token=…`). Токен берётся из `api_auth.token` либо автоматически генерируется при запуске в `conf/api-token` (права 0600). Команда `citeck ui` печатает — и открывает — аутентифицированную ссылку. Статические файлы интерфейса остаются общедоступными; защищается только API. Unix-сокет, десктопное приложение и клиенты с mTLS не затрагиваются.

(Этот раздел — про админский интерфейс *лаунчера*, а не про интерфейс платформы Citeck, в который вы входите после установки.)
